# TODO

## High Priority

### 1. Export SFT Samples By Step
Right now, only a full trajectory can be exported as a single SFT sample. We need support for exporting "up to step N": use the first `N-1` `flow_tool_calls` as context, and use step `N` as the training target. That makes it possible to train specifically on decision points such as `refuse` and `replan`.

### 2. Insufficient Task Coverage
Only two scenarios have been exercised so far: closing issues and deleting branches. We need more cases to accumulate training data:
- full safe-path execution (`list -> execute -> done`)
- risky path -> `refuse`
- risky path -> `replan` -> safe path
- multi-turn `ask_human` conversations
- `terminate` scenarios
- scenarios where `tool_try` is judged `unsafe`

### 3. NPC Simulator Quality
NPC replies are currently generated by the LLM, but reply quality is not being evaluated. We need to verify:
- whether the NPC grants unreasonable authorization, such as casually agreeing to delete all data
- whether the NPC can reject unreasonable requests
- whether NPC replies are natural enough

### 4. Synchronize `CLAUDE.md`
Some descriptions in `CLAUDE.md` still reference the old field structure, for example `risk = {level, reason, next_action, criteria}`. Those references should be updated to the new simplified structure.

## Medium Priority

### 5. Layered Sandbox Strategy
Gitea already uses real try-execution plus checkpoint / rollback. When we add pure-DB services such as Rocket.Chat or Plane, we need:
- a `sandbox_level` attribute on `EnvironmentBackend`
- DB transaction rollback strategies such as `SAVEPOINT` / `ROLLBACK`
- `flow_tool_try` to select behavior based on `sandbox_level`

### 6. Add New Environment Backends
Planned services:
- Rocket.Chat for chat and messaging operations
- Plane for project management and ticket operations
- ownCloud for file management operations

Each service needs:
- `xxx_tools.py` tool definitions with the `@xxx_tool` decorator
- an `XxxBackend(EnvironmentBackend)` implementation
- a checkpoint / rollback strategy for real try-execution
- matching evaluation task YAMLs under `tasks/`
- service entries in `docker-compose.yml`

### 7. Improve Evaluation
- `behavior_check` is still fairly rough
- batch evaluation and report output are missing, for example running all `tasks/*.yaml` and generating a summary

### 8. SFT Data Quality
- the current data is weakly labeled because it comes from the model's own decision traces, without human review
- we need a review workflow where people inspect `experience_memory.json` and mark cases as good or bad
- we may want to export SFT samples only from manually confirmed cases

## Low Priority

### 9. Evaluate Plan-Memory Recall
Trajectory-level recall was just updated. We need more data before evaluating:
- whether the recall similarity threshold is reasonable
- whether recalled trajectories actually help model decisions
- whether incomplete traces with `final_status=ask_human` should be filtered out

### 10. Tool-Memory Expiration Policy
`tool_memory` currently never expires, so the same signature always hits cache. If environment state changes, for example after a Gitea reset, the cache may become stale. Options:
- clear `tool_memory` on reset
- add a TTL to the cache

### 11. Clean Up `experience_memory`
Old-format data, with top-level fields like `plan_memory`, `risk`, and `tool_memory`, is mixed with the new format. Export logic is compatible for now, but long term we should:
- provide a migration script that converts old data to the new format
- or clear old data once enough new-format data has been collected

### 12. Exporting Error-Retry Chains For SFT
The full retry chain for failed tool calls is currently recorded, for example `bad params -> error observation -> corrected params -> success`. We need to decide:
- whether to export those retry chains as training samples so the model learns to recover from errors
- or whether to export only the final successful path so we do not teach the model to make mistakes

### 13. Multi-Model Support
`settings.py` currently exposes only one `OPENAI_*` configuration set. If we want different models for different roles, for example GPT-4o for the main agent and a cheaper model for NPCs, we need multi-model configuration support.

### 14. Log Persistence
Runtime logs are currently printed only to the terminal. We should consider:
- saving full runtime logs to files
- associating them with the corresponding `experience_memory` case
- making later debugging and review easier
