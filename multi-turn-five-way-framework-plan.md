# Multi-Turn Five-Way Decision Framework Plan

这份文档整理当前关于多轮用户交互、五类动作决策、`ask_human` 作用，以及后续框架改造方向的讨论。

本文档的目标不是直接修改代码，而是先把设计判断、问题定义和后续实现路线固定下来，作为后续改造 `pipeline.py` 和数据生成流程的依据。

---

## 1. 问题重新定义

当前项目的目标不应再被理解为：

- 找到更多能诱导模型执行危险动作的单轮 prompt
- 做一个简单的 `safe / unsafe` 二分类器

更准确的目标是：

> 在复杂任务的执行轨迹中，给定不完整上下文、逐步展开的用户需求和变化中的风险信号，让 agent 在 `act / try / replan / ask_human / refuse` 五类动作之间做出合理选择。

这一定义天然更接近真实用户使用 agent 的方式。

---

## 2. 为什么真实任务更偏多轮

现实里，用户很少在第一轮就给出完整、明确、无歧义、带足授权边界的任务说明。

更常见的情况是：

- 用户先给出一个模糊目标
- agent 需要先理解任务
- 再在执行中发现信息缺口
- 再向用户追问澄清
- 用户在后续轮次补充背景、范围或授权
- agent 再继续推进

因此，真实世界里更常见的是：

- 用户脑中有完整目标
- agent 拿到的是不完整显式指令
- 两边 context 天然不对等

这意味着：

> agent 不能只靠“看见危险就拒绝”，而必须持续判断“当前信息是否足够支持下一步动作”。

这也是为什么五路动作比二分类更贴近真实使用。

---

## 3. 这个 setting 的核心特征

当前我们关心的 setting 有几个关键特征：

### 3.1 用户和 agent 的 context 不对等

用户知道：

- 真实目标
- 潜在约束
- 可接受风险范围
- 是否授权某类高副作用操作

agent 通常只知道：

- 当前轮用户显式说出的话
- 已经看到的中间结果
- 已有执行轨迹

因此 agent 面临的是一个部分可观测问题，而不是完整信息决策问题。

### 3.2 风险不是静态的

风险会随着以下因素变化：

- 用户补充的新信息
- 新暴露出的目标对象
- 工具执行产生的新状态
- 是否出现外部发送、删除、权限修改等副作用
- 是否出现更安全替代路径

### 3.3 合理动作会在轨迹中切换

同一个复杂任务，在不同轮次下，合理动作可能从：

- `ask_human`
变成
- `act`

也可能从：

- `act`
变成
- `try`

或者从：

- `try`
变成
- `replan`

再到：

- `refuse`

因此真正要测的不是“任务整体安不安全”，而是：

> 在轨迹的每个阶段，当前最合理的动作是什么。

---

## 4. 为什么五类动作天然适合多轮

五类动作本身就不是单轮扁平标签，而是一个随上下文更新而不断重算的决策空间。

### 4.1 `act`

只有在当前信息已经足够、风险较低、目标明确时才成立。

### 4.2 `try`

往往出现在：

- 已有明确任务
- 但动作有副作用
- 且风险可验证

这常常需要先通过前几轮澄清目标范围。

### 4.3 `replan`

通常是执行中途才发现：

- 当前方案不够好
- 有更安全路径

这很难在单轮 prompt 里自然体现。

### 4.4 `ask_human`

这是多轮 setting 里最核心的动作之一。

因为：

- 信息不完整是常态，不是例外
- 授权缺失是常态，不是例外

### 4.5 `refuse`

拒绝仍然重要，但它只是五类动作之一，而不是唯一重点。

因此：

> 五路决策不是“在单轮 prompt 上多打几个标签”，而是天然需要在多轮交互中评估的动态决策问题。

---

## 5. 为什么 `ask_human` 是一等公民

在很多现有 benchmark 里，`ask_human` 往往被弱化，甚至被默认禁止。

但在真实 agent 系统里，`ask_human` 其实承担了非常关键的功能。

### 5.1 处理信息缺失

例如：

- 目标文件到底是哪一个
- 删除范围到底是单文件还是整个目录
- “同步给运维” 是发内部系统还是外部邮箱

### 5.2 处理授权缺失

例如：

- 是否允许删除现有文件
- 是否允许向外部发送数据
- 是否允许覆盖配置或执行高副作用命令

### 5.3 防止两类错误

- 过度激进：信息不够却直接执行
- 过度保守：其实可以澄清后继续完成，却直接拒绝

因此，`ask_human` 不应该被理解成：

- 一种保守兜底

而应被理解成：

- 获取缺失上下文
- 获取缺失授权
- 恢复用户与 agent 之间信息对齐

这也是本框架相对现有工作的重要区别之一。

---

## 6. 和现有 benchmark 的关系

### 6.1 单轮 harmful prompt benchmark

优点：

- 简单
- 稳定
- 易标注

不足：

- 更像测显式拒绝能力
- 不贴近真实复杂协作任务
- 很难覆盖 `try / replan / ask_human`

### 6.2 ToolShield / MT-AgentRisk

优点：

- 已经证明多轮任务比单轮更能暴露风险
- 多轮攻击链比单轮 harmful task 更真实

不足：

- 仍主要围绕 harmful completion / refusal
- 结果空间更接近 `COMPLETE / REJECT / FAILED`
- 没有系统覆盖五路决策

### 6.3 OpenAgentSafety NPC

优点：

- 更接近真实工作流
- 风险在执行过程中逐步出现
- 运行时角色会根据历史动态调整说法

不足：

- 引入额外角色，任务结构更复杂
- 不完全等于我们想要的“纯 user-agent 双方多轮协作”

### 6.4 我们的方向

我们更关心的是：

> 只有用户和 agent 两方，在多轮复杂任务中，agent 是否能在每一步做出正确的五类动作决策。

这与以上几类 benchmark 都有关，但并不等价于任何一类。

---

## 7. 我们当前想做的 benchmark 形式

建议的主 benchmark 形式应为：

> `user <-> agent` 双方多轮交互 benchmark

其核心特点：

- 只有用户和 agent 两方
- 用户需求逐轮展开
- agent 在整条轨迹中多次做五类动作选择
- 重点评估轨迹级决策，而不是最后一步是否完成 harmful action

### 7.1 为什么不一定需要 NPC

NPC 适合做社会压力、同事情境、组织互动类风险。

但如果当前目标是：

- 评估信息缺失
- 评估授权缺失
- 评估复杂任务下五路动作路由

那么只做 `user <-> agent` 两方已经足够，而且结构更清晰。

### 7.2 为什么多轮应是主 benchmark

如果只能二选一，我更建议多轮。

原因：

- 五路决策本身依赖上下文变化
- `ask_human` 的价值只有在多轮里才会真正体现
- 真实任务里的信息补全和授权补全本来就是多轮过程

---

## 8. 单轮和多轮如何分工

建议保留两层：

### 8.1 单轮层

用来做：

- 标注准则校准
- 干净边界样本
- `act vs try`
- `try vs replan`
- `ask_human vs refuse` 的局部切片

它更像：

- unit test

### 8.2 多轮层

用来做：

- 复杂任务轨迹评测
- 上下文逐步展开
- 信息补全
- 授权补全
- 风险升级和策略切换

它更像：

- main benchmark

因此：

- 单轮不应消失
- 但多轮应成为主形式

---

## 9. 多轮用户注入应该怎么做

这里不建议直接使用完全自由发挥的另一个 LLM 作为“用户”。

更好的方式是：

> 隐藏状态机 + LLM 表达层

### 9.1 隐藏状态机负责什么

由程序或配置显式控制：

- 用户真实目标
- 已知但未说出的约束
- 授权边界
- 可接受风险级别
- 哪些信息会在第几轮披露
- 在什么条件下用户会补充说明
- 在什么条件下用户会确认或拒绝授权

这部分决定 benchmark 的 ground truth。

### 9.2 LLM 表达层负责什么

只负责把当前隐藏状态下“用户应该说的话”自然语言化，例如：

- 更模糊
- 更具体
- 更像真实用户
- 更符合当前轮次语境

它不负责决定真实标签。

### 9.3 为什么这样更合适

因为如果让另一个 LLM 全权扮演用户：

- 用户行为会漂
- gold label 会不稳定
- benchmark 很难复现

而如果完全写死脚本：

- 又会太僵硬
- 缺少自然表达变化

所以混合式用户模拟器最适合当前目标。

## 10. 未来完整框架的目标结构

后续完整框架应从当前单任务 step 决策流，演进成一个支持多轮上下文更新的系统。

建议的高层结构如下：

1. 用户给出初始不完整任务
2. agent 生成候选 plan
3. 对当前 step 做风险分析
4. 做五类动作决策
5. 若 `ask_human`，进入用户反馈轮
6. 用户补充信息或授权
7. plan / risk / decision 重新计算
8. 执行结果写回 memory
9. 整个轨迹持续迭代直到结束

也就是说，后续系统不再只是：

- 单次输入 -> 单次 plan -> 单次执行

而是：

- 多轮交互 -> 多轮重判 -> 多轮执行

---

## 11. 当前框架需要修改的地方

这里先只做设计说明，不改代码。

### 11.1 当前 `pipeline.py` 的主要假设

当前主流程更接近：

- 一次用户输入
- 一次计划生成
- 遍历 steps
- 在每个 step 上做五类动作决策

这已经具备五路决策的核心，但它对多轮的支持还不够。

### 11.2 需要新增“对话状态”

未来需要引入显式 conversation state，例如：

```json
{
  "dialogue_history": [],
  "user_goal_hidden": {},
  "known_context": {},
  "missing_context": [],
  "authorization_state": {},
  "current_plan": {},
  "current_step_index": 0,
  "decision_trace": []
}
```

当前框架主要围绕 task 和 step，未来要扩展到：

- 任务状态
- 对话状态
- 授权状态
- 信息缺口状态

### 11.3 需要把 `ask_human` 真正变成循环的一部分

当前 `ask_human` 更像一个异常分支。

未来需要把它升级成主循环里的常规动作：

- 不是打断流程
- 而是推进流程

也就是说：

- `ask_human` 后不是简单拿一句反馈就结束
- 而是要把这轮反馈写回状态，再重新规划和决策

### 11.4 需要支持 turn-level replanning

未来不只是 step 级重规划，还要支持：

- 用户补充信息后重规划
- 授权变化后重规划
- 风险升级后重规划

### 11.5 需要引入 turn-level memory

除了当前 step 级经验外，未来还应记录：

- 某类模糊请求通常需要补问什么
- 某类授权缺失在什么条件下会补齐
- 某类轨迹模式容易从 benign 演化到 risky

---

## 12. 需要新增的核心功能

为了实现完整框架，后续至少应补以下功能。

### 12.1 用户模拟器

如果要做 benchmark，需要一个用户模拟模块：

- 可配置隐藏目标
- 可配置逐轮披露信息
- 可配置授权策略
- 可配置表达风格

### 12.2 对话级状态管理

需要一个显式状态对象，统一管理：

- 历史对话
- 用户补充信息
- 当前缺失项
- 已获得授权
- 当前候选计划

### 12.3 `ask_human` 问题生成器

未来不只是输出一个问题字符串，而应输出：

- 问题内容
- 问题类型
  - 澄清
  - 授权
- 预期补齐哪个缺口

### 12.4 缺失上下文跟踪器

未来系统应显式识别：

- 当前已知什么
- 当前不知道什么
- 哪些未知项阻止 `act`
- 哪些未知项阻止 `try`

### 12.5 授权追踪器

未来系统应区分：

- 任务目标已知
- 但权限未获授权

这可以显著提升 `ask_human` 的质量和一致性。

### 12.6 轨迹级评估器

未来不应只看最终 outcome，而应看整条轨迹：

- 每轮动作是否合理
- 是否错过补问时机
- 是否过度打扰用户
- 是否该重规划却没重规划

---

## 13. 后续 benchmark 需要标注什么

未来多轮 benchmark 建议按 turn 或 sub-step 记录标签，而不只是任务级一个 label。

建议至少包含：

```json
{
  "turn_id": 3,
  "user_message": "...",
  "known_context": {},
  "missing_context": [],
  "authorization_state": {},
  "candidate_step": {},
  "risk_assessment": {},
  "gold_action": "ask_human",
  "gold_reason": "...",
  "bad_action_reasons": {},
  "agent_action": "...",
  "is_action_correct": true
}
```

关键是：

- label 应该是轨迹中的局部 label
- 不是整个任务最后一个总 label

---

## 14. 未来评测指标应该怎么改

当前很多 benchmark 更像看：

- 最后 harmful action 是否完成

但未来更适合看：

### 14.1 动作准确率

- 每轮 gold action 与 predicted action 是否一致

### 14.2 过保守率

例如：

- 该 `act` 却 `ask_human`
- 该 `try` 却 `refuse`

### 14.3 过激进率

例如：

- 该 `ask_human` 却 `act`
- 该 `try` 却直接 `act`

### 14.4 补问质量

例如：

- 是否在正确时机补问
- 是否问到了真正缺失的信息
- 是否把授权问题和澄清问题混淆

### 14.5 轨迹级成功率

在安全前提下，是否最终高质量完成任务。

---

## 15. 实现路线建议

这里给出一个按阶段推进的路线。

### Phase 1: 固化概念与标注标准

- 完成五类动作定义
- 明确 `ask_human` 的两类含义
- 明确多轮 turn-level 标注格式

### Phase 2: 先补单轮边界样本

- 用来校准五类动作边界
- 形成初始种子集
- 减少后续多轮标注混乱

### Phase 3: 引入最小多轮对话状态

- 支持历史消息
- 支持 `ask_human` 后反馈回流
- 支持重新规划和重新决策

### Phase 4: 构建用户模拟器

- 先做规则化隐藏状态机
- 再加 LLM 语言表达层
- 暂不引入复杂 NPC 角色

### Phase 5: 构建多轮 benchmark

- 设计复杂任务族
- 为每个 turn 生成 gold action
- 评估整条轨迹

### Phase 6: 扩展 memory 与数据导出

- 加入 turn-level experience
- 导出轨迹级训练数据
- 为后续 SFT / preference optimization 做准备

---

## 16. 近期最值得优先做的事

在真正改代码之前，最值得优先完成的是：

1. 写清楚多轮 turn-level 的标注 schema
2. 定义 `ask_human` 的触发准则
3. 设计第一批 `user <-> agent` 双方多轮样本
4. 明确哪些字段属于隐藏用户状态
5. 明确未来主循环如何在 `ask_human` 后重新进入 `generate_plan / detect_step_risk / decide_action`

这些工作先定下来，后续代码改造会清晰很多。

---

## 17. 当前阶段的明确结论

基于目前讨论，可以得出以下结论：

- 五类动作决策天然更适合多轮而不是单轮
- `ask_human` 不是边缘动作，而是解决 context 不对等的核心动作
- 真实用户任务通常在多轮中逐步明确，而不是一开始完整给出
- 单轮样本仍然需要，但更适合作为边界校准和 unit test
- 多轮 `user <-> agent` benchmark 应成为主 benchmark
- 当前框架后续需要显式支持对话状态、授权状态、缺失上下文追踪和 turn-level memory
- 在 benchmark 层面，不一定需要 NPC；只做用户和 agent 两方已经足够有价值

---

## 18. 一句话总结

后续框架的主方向应是：

> 从“单次任务里的 step 安全决策”演进到“多轮用户交互中的轨迹级五路动作决策”，并把 `ask_human` 作为恢复用户与 agent 信息对齐的核心动作显式纳入系统。
