 Context                                                                       
                                                                                                                                          
     当前 8 个服务各有 ~10-12 个工具，按 is_write: bool 分类。扩充到 35-40                                                                
     个工具后需要解决：
     1. 工具发现 — 当前 system prompt 没有注入 real tool 列表，模型靠 risk
     guidance 文字和历史轨迹"猜"工具名。扩充后不可行
     2. Token 开销 — 35 工具完整 schema ≈ 8000-15000 tokens，每轮重复传递不现实
     3. 分布均匀性 — 每个服务 "5 读 + 7 写" 不反映真实 API

     约束：pipeline 流程不变 — predict_risk 保持 safe/risky，phase
     路由、memory、SFT 导出均不改。

     ---
     一、整体方案：三层工具呈现架构

     ┌─────────────────────────────────────────────────┐
     │  Layer 1: 领域工具组 (Domain Tool Groups)         │
     │  始终在 system prompt / snapshot 中               │
     │  每组一行：组名 + 描述 + 包含的操作类型            │
     │  8-12 组 ≈ 50-80 tokens                          │
     ├─────────────────────────────────────────────────┤
     │  Layer 2: 语义检索 Top-K 工具索引                  │
     │  根据当前 task + step context 检索                │
     │  返回 top-k 个候选工具的 name + 简短描述           │
     │  k=8~12 ≈ 50-100 tokens                         │
     ├─────────────────────────────────────────────────┤
     │  Layer 3: 当前工具完整 Schema                     │
     │  模型在 predict_risk 选定工具后                    │
     │  need_try / need_real_tool 阶段注入该工具的完整参数 │
     │  1 个工具 ≈ 150-300 tokens                       │
     └─────────────────────────────────────────────────┘

     总计每轮：~250-480 tokens（vs 不优化的 8000-15000）

     三、Layer 2: 语义检索 Top-K 候选（检索层）

     3.1 思路

     利用项目已有的 FAISS + sentence-transformers 基础设施（memory.py 中的
     PlanMemoryVectorStore），构建一个工具语义索引。每轮根据当前 task + step
     context 检索 top-k 最相关的工具，只把这些候选放入 snapshot。

     这对应你搜索到的方案 1。

     3.2 实现

     新文件: safety_pipeline/tool_retrieval.py

     class ToolIndex:
         """基于 FAISS 的工具语义检索索引"""

         def __init__(self, registry: ServiceToolRegistry):
             self.registry = registry
             self.index = None        # FAISS index
             self.tool_names = []     # 与 FAISS index 行对齐的工具名列表

         def build_index(self):
             """为所有工具的 description 构建 embedding 索引"""
             texts = []
             for tool in self.registry._tools.values():
                 # 拼接工具名 + 描述 + 组名作为检索文本
                 text = f"{tool.name}: {tool.schema['function'].get('description',
     '')} [group: {tool.group}]"
                 texts.append(text)
                 self.tool_names.append(tool.name)

             embeddings = self._embed_texts(texts)  # 复用 memory.py 的 embedding
     逻辑

             import faiss
             dim = embeddings.shape[1]
             self.index = faiss.IndexFlatIP(dim)  # 内积相似度（已归一化 = 余弦）
             self.index.add(embeddings)

         def retrieve(self, query: str, top_k: int = 10) -> list:
             """根据查询语义检索最相关的 top_k 个工具"""
             query_vec = self._embed_text(query)
             scores, indices = self.index.search(query_vec.reshape(1, -1), top_k)
             results = []
             for score, idx in zip(scores[0], indices[0]):
                 if idx < 0:
                     continue
                 tool_name = self.tool_names[idx]
                 tool = self.registry._tools[tool_name]
                 results.append({
                     "name": tool_name,
                     "desc": tool.short_description or
     tool.schema["function"]["description"][:80],
                     "group": tool.group,
                     "is_write": tool.is_write,
                     "score": float(score),
                 })
             return results

     3.4 降级策略

     FAISS/sentence-transformers 是可选依赖。如果不可用：
     - 降级为 BM25 关键词匹配（用 task 文本中的关键词匹配工具 name +
     description）
     - 或直接降级为返回当前组的全部工具（回退到 Layer 1）

     这与现有的 PlanMemory 降级策略一致（faiss-cpu is optional; plan memory degrades  gracefully）。

     def build_agent_state_snapshot(state):
         snapshot = {
             "user_task": state["initial_user_input"],
             # ... 现有字段 ...
         }

         # Layer 1: 工具组（始终注入）
         snapshot["tool_groups"] = tool_index.registry.get_tool_groups_summary()

         # Layer 2: 语义检索候选（选工具阶段注入）
         if state["flow_phase"] in ("need_risk", "need_step", "need_next_or_done"):
             query = compose_tool_retrieval_query(state)
             snapshot["candidate_tools"] = tool_index.retrieve(query, top_k=10)

         # Layer 3: 当前工具完整 schema（执行阶段注入）
         if state["flow_phase"] in ("need_try", "need_real_tool"):
             current_step = get_current_step(state)
             if current_step:
                 tool_name = current_step.get("tool", "")
                 schema = real_tool_schema_map.get(tool_name)
                 if schema:
                     snapshot["current_tool_schema"] = schema

         return snapshot

     七、与现有 Memory 系统的协作

     7.1 PlanMemory → 辅助检索

     memory_for_plan 召回的历史轨迹中包含曾用过的工具名。可以用这些工具名补充
     Layer 2 的检索候选（boost 历史用过的工具）：

     def compose_tool_retrieval_query(state):
         """组合检索查询，融合 task + plan memory"""
         parts = [state["initial_user_input"]]
         # 从 plan_memory 中提取历史使用的工具名
         plan_mem = state.get("current_plan_memory", {})
         for traj in plan_mem.get("trajectories", []):
             for step in traj.get("steps", []):
                 tool = step.get("tool", "")
                 if tool:
                     parts.append(tool)
         return " ".join(parts)

     7.2 ToolMemory → 不变

     ToolMemory 在 predict_risk 之后自动执行，按工具名精确匹配。不受检索层影响。
