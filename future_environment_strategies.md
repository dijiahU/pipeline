# Agent Execution Environment Strategies: Tackling State & Dependency Issues

在 Agent 的 Safety Pipeline (Plan + Execution 双重验证框架) 中，确保 **“沙箱试运行环境 (Try Sandbox)”** 与 **“真实业务环境 (Real Environment)”** 的状态绝对一致，是决定框架防御有效性的核心。

如果 Agent 的操作是**环境依赖型**的（例如需要读取特定的 `.env` 文件，或者依赖前几个 Step 下载的临时文件），而在执行安全校验时每次都拉起一个全新的“空沙箱”，会导致：
1. **安全校验失效（假阴性）：** 沙箱里没有关键文件，危险操作（如窃取密码）无法暴露，判定为 Safe。
2. **正常任务阻断（假阳性）：** 沙箱里缺少依赖库或前置文件，正常命令执行报错，导致任务意外终止。

为了解决这一问题，指导未来的工程演进，我们整理了两种核心解决方案：**静态预构建模板法** 与 **动态运行时克隆法**。

---

## 方案一：静态预构建模板法 (Static Template Injection)

此方案的核心思想类似于“烧录镜像”。在 Agent 会话开始前，将所需的所有环境依赖（文件、系统包、Python 库等）提前打包成一个固定的模板。

### 1. 实现方案与准备工作
*   **依赖准备：**
    *   梳理当前任务所需的静态依赖（例如包含假密码的 `.env` 模拟文件、测试用的数据库 `.sqlite`、前置代码文件）。
    *   编写配置文件（如 `e2b.toml` 或 Dockerfile）。
*   **构建阶段 (Build Time)：**
    *   利用 E2B CLI 等工具，基于配置文件构建一个自定义模板（例如命名为 `task-env-v1`）。
*   **运行时调用 (Runtime)：**
    *   修改 Pipeline 代码，在 `tool_try` 或正式执行拉起沙箱时，通过传入模板参数（如 `Sandbox.create(template="task-env-v1")`）直接获得一个包含预置文件的环境。
    *   *退阶平替实现：* 若不使用镜像构建，也可在每次 `Sandbox.create()` 之后，立刻通过 Python 代码 (`sandbox.files.write(...)`) 注入一组 Mock 文件。

### 2. 设计初衷与适用场景
*   **为什么这么设计？** 实现成本极低，完全依赖现有云厂商（如 E2B）的基础设施 API，无需深入底层虚拟化。
*   **适用场景：**
    *   **基准测试 (Benchmarks)：** 运行大量已知状态的静态测试集（如 SWE-bench）。
    *   **无状态/单步任务：** 任务本身不产生复杂的中间状态，对环境的依赖仅限于初始数据。
*   **局限性：** 无法应对“状态漂移”。如果 Agent 在 Step 1 修改了环境，由于每次 `tool_try` 都是拉起初始模板，Step 2 的 `tool_try` 将丢失 Step 1 的状态变更。

## 方案二：动态运行时克隆法 (Master-Shadow Snapshot Cloning)

此方案是未来 Agent 基础设施的终极形态，其核心思想类似于“游戏存档与读档”或 Linux 的 `fork()` 机制。

### 1. 实现方案与准备工作
*   **架构准备：**
    *   整个 Agent 会话生命周期中，仅存在**唯一一个主沙箱 (Master Sandbox)**。
    *   所有已确认安全的操作 (Safe Tool Calls) 都在此主沙箱中执行，状态（磁盘文件、内存变量）持续累积。
*   **运行时调用 (Runtime & Forking)：**
    *   当 Pipeline 需要验证一个高风险或未知的 Tool Call 时，**并非**创建一个全新的空沙箱，而是向底层虚拟化引擎（如基于 Firecracker 的 MicroVM）发送请求。
    *   引擎在毫秒级时间内，为主沙箱的当前内存和磁盘打一个热快照 (Live Snapshot)，并利用 **Copy-on-Write (写时复制)** 技术瞬间克隆出一个 **影子沙箱 (Shadow Sandbox)**。
    *   将该待验证的 Tool Call 发送至影子沙箱执行。
*   **裁决与回收：**
    *   在影子沙箱执行完毕后，执行安全校验（对比快照前后的关键状态变更）。
    *   如果判定为 **危险**：直接销毁影子沙箱，主沙箱不受任何污染，将错误返回给 LLM 重新 Plan。
    *   如果判定为 **安全**：销毁影子沙箱，在主沙箱中正式执行该动作（或若支持状态合并，直接合并状态）。

### 2. 设计初衷与适用场景
*   **为什么这么设计？** 解决连续交互式 Agent 面临的“薛定谔的状态”难题。在 Computer Use（计算机控制）或复杂的长流程代码重构任务中，Agent 的下一步动作重度依赖上一步留下的内存变量或临时文件。只有内存级别的快照克隆，才能让“试错环境”与“真实环境”达成 100% 的严格一致，且避免了从头重新执行造成的巨大时间开销。
*   **适用场景：**
    *   **长上下文持续执行：** 多步骤的复杂工程任务。
    *   **Computer Use Agent：** 图形界面的自动化操作（每一次点击都改变了屏幕 UI，出错后必须恢复到点击前的 UI 快照状态）。
*   **性能预期：** 结合 MicroVM 技术，克隆一个携带完整状态的影子沙箱耗时可压缩至 `10ms - 50ms` 级别，几乎等于一次轻量级 API 调用。

## 未来工作方向建议

对于本项目的演进，建议分为两步走：
1.  **近期 (MVP 巩固)：** 在测试代码中实装“静态预构建法”的轻量化版本（即 `setup_mock_env` 函数），在每次创建沙箱后自动注入几个包含敏感信息的模拟 `.env` 和 `.log` 文件，跑通针对文件读写的安全拦截真实流程。
2.  **远期 (架构升级)：** 持续关注 E2B 或其他沙箱供应商是否推出原生的 "Snapshot/Fork" API。一旦该功能就绪，立即将 Execution 阶段从“并行独立创建”重构为“主控-影子克隆”模式，为进军极高自由度的 Computer Use 场景打下基础设施地基。
