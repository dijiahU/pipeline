# 服务扩展实施指南

本文档用于复盘本轮 Gitea 扩展改造，并作为后续 7 个服务继续扩工具、补任务、做评测的统一实施模板。

目标不是重复 [工具改动方案.md](/Users/rick/Desktop/pipline/pipeline/工具改动方案.md) 的概念设计，而是把“已经在代码里跑通的做法”和“真实踩过的坑”整理成可执行清单。

---

## 1. 目标与口径

当前项目的扩展目标有两层：

1. 共享基础设施层
   为所有服务提供三层工具呈现能力：
   - Layer 1: `tool_groups`
   - Layer 2: `candidate_tools`
   - Layer 3: `current_tool_schema`

2. 服务落地层
   对每个服务完成：
   - 工具注册元数据补齐
   - 工具数量扩充
   - seed / baseline 补齐
   - outcome / try summary 补齐
   - 任务 YAML 扩充
   - live validation

本轮 Gitea 达成的是“场景覆盖完成”，不是严格的“1 工具 = 1 YAML”覆盖。

当前 Gitea 状态：
- 工具数：`41`
- 工具组：`10`
- 任务 YAML：`39`
- 覆盖口径：读工具适度打包，写工具尽量单测

## 3. 后续服务统一实施模板

后续 7 个服务建议严格按下面顺序推进，避免“工具写完了但 seed 不够 / 任务写了但评测跑不通”。

### Step 1. 先定义工具组

先在 [service_tools.py](/Users/rick/Desktop/pipline/pipeline/safety_pipeline/service_tools.py) 为目标服务补 `_SERVICE_TOOL_GROUPS`。

要求：
- 每个服务控制在 `8-12` 个组以内
- 组名按领域划分，不按 HTTP endpoint 划分
- 组描述直接面向模型，不要写成开发者注释
- 每组尽量同时含读/写语义边界，方便模型先决定“在哪个域里找工具”

经验：
- 工具组质量会直接影响 `candidate_tools` 检索质量
- 组名不要太泛，例如 `general_ops` 没价值

### Step 2. 扩工具前先定 seed 对象

不要先写工具，再想评测。

先确定：
- 该服务需要哪些固定对象才能稳定评测
- 哪些对象必须预置
- 哪些对象应当在 task 中临时创建再删除

Gitea 的经验是：
- collaborator 相关工具如果没有固定二号用户，任务很难稳定
- 只靠管理员一个账号，很多权限类任务会变成“纸面覆盖”

### Step 3. 工具实现要一次性补元数据

每个工具实现时同时补：
- `group`
- `short_description`
- 精确参数 schema

不要留到后面补。

经验：
- 模型最容易错在 `predict_risk.tool_args` 缺字段
- 如果 `short_description` 太模糊，检索层会把相邻工具混在一起

### Step 4. environment 同步补 try / outcome

任何新增写工具，都要同步检查：

1. `run_try()` 是否能真实执行
2. `try summary` 是否有 meaningful affected object
3. 是否需要新的 `outcome_check.condition`

典型情况：
- `branch_exists`
- `collaborator_exists`
- “对象已删除 / 已创建 / 状态已恢复”

### Step 5. 任务 YAML 先做“强覆盖”而不是“绝对逐工具”

建议策略：
- 读工具：按领域打包
- 写工具：尽量 `1 工具 1 任务`
- 生命周期类工具：允许 `创建 -> 删除` 合并成一个任务

推荐任务规模：
- 每个服务先做到 `24-32` 个 YAML
- 如果要追求严格逐工具，再往 `35-40+` 扩

### Step 6. 文案必须给出显式参数

任务文案不要只写目标，要尽量写出：
- 项目名 / 仓库名 / 表名 / ticket 编号 / topic id
- 用户名 / 文件名 / 分支名 / tag 名
- 必要的正文、颜色、时间、权限级别、公钥内容

经验：
- 文案不显式，模型就会漏 `tool_args`
- schema hint 虽然能救一部分，但不能保证稳定 pass

### Step 7. 验证必须分三层

每个服务都至少做：

1. 静态验证
   - `py_compile`
   - YAML parse
   - `--list-service-tools`
   - `--list-service-tasks`

2. live API/tool 验证
   - 对核心读工具做真实调用
   - 对核心写工具做 create / delete / rollback 验证

3. evaluation 验证
   - 读任务至少抽 2-3 条
   - 写任务至少抽 2-3 条
   - 拒绝任务至少抽 1-2 条

## 4. Gitea 复用结论

Gitea 提供了后续服务最重要的 5 条经验。

### 4.1 宽任务比窄任务更容易失败

例如：
- “列出仓库，再看设置，再看目录，再看 tag” 这种 overview 任务

模型容易：
- 选错第一步工具
- hallucinate 不存在的工具名
- 漏 `tool_args`

因此：
- 每个服务都应该保留少量 overview 任务
- 但不要指望 overview 任务是最稳定的 smoke test
- smoke test 应优先选窄读任务

### 4.2 权限类写操作容易被判成 risky

Gitea 的 `add_collaborator` 就出现了这个问题：
- task 文案已经说“已获确认”
- 模型仍然把“授予 write 权限”判成 risky
- 最终走到了 `ask_human`

结论：
- 权限类操作要单独看待
- 对后续服务里的用户角色、分享权限、管理员动作，不要默认它们会自动落到 `safe`

### 4.3 评测不要并行跑同一个服务

Gitea 是单容器 reset + reseed 模式。

并行跑两个 evaluation 时，会出现：
- reset 互相覆盖
- seed 半途冲突
- outcome_check 被污染

这个结论适用于几乎所有当前服务。

建议：
- 同一服务评测串行跑
- 每条写任务前先 reset

### 4.4 memory 只能证明“曾经有轨迹”，不能证明“评测通过”

`memory/experience_memory.json`、`tool_memory.json`、`sft_dataset.jsonl` 可以证明：
- 某条任务曾成功走过若干 step
- 某个工具已有 safe case

但不能严格替代 evaluator 的 `PASSED/FAILED`。

### 4.5 seed 是工具覆盖的一部分，不是附属品

没有稳定 seed，对很多工具来说根本无法构造可靠任务。

Gitea 的 collaborator 工具直到加了固定用户后才真正可测。

## 5. 后续 7 个服务逐个注意事项

下面只写“后续扩工具时真正需要额外小心的点”。

### 5.1 Rocket.Chat

重点方向：
- channels
- messages
- users
- integrations / webhooks / bots

特殊注意：
- reset 依赖 MongoDB volume，评测不要并发跑
- 频道、私聊、私有组是不同模型，工具设计不要混
- 消息删除、频道归档、用户删除很容易被模型判成 risky
- 如果要扩 webhook / bot 类工具，必须额外准备“内部地址”和“外发地址”两类任务
- seed 最好补齐：
  - 公共频道
  - 私有频道
  - 至少两个普通用户
  - 一条 DM / 一条线程型消息

建议新增工具组：
- `channels`
- `messages`
- `users`
- `integrations`

### 5.2 ownCloud

重点方向：
- file browse
- file ops
- sharing

特殊注意：
- ownCloud 现在走的是 oCIS，默认 `https://localhost:9200`，很多请求要处理自签名 TLS
- WebDAV 返回 XML，路径规范化、尾部 `/`、URL encode 很容易出错
- 分享链接和用户共享是两套语义，不要混在同一工具里
- 文件移动/复制/删除任务一定要写清楚源路径和目标路径
- 如果要扩公开分享任务，要区分：
  - 内部共享
  - 无密码公开链接
  - 带权限变更的公开链接

建议 seed 补齐：
- 多层级目录
- 至少 1 个共享文件
- 至少 1 个公开链接
- 一组敏感文件与一组普通文件

### 5.3 NocoDB

重点方向：
- schema
- row CRUD
- bulk operations

特殊注意：
- NocoDB 的对象标识常常是 `base_id` / `table_id` / `record_id`，任务文案最好显式给业务名和主键
- schema 工具和 record 工具不要复用同一组文案，模型容易混
- 批量删除 / truncate 类操作非常适合做 refuse / ask_human
- 某些记录更新任务如果只给自然语言字段名，模型容易漏主键
- setup / reset 依赖 PostgreSQL 重建，批量评测时间会更长

建议新增工具组：
- `schema`
- `records`
- `views_queries`
- `bulk_ops`

### 5.4 Zammad

重点方向：
- tickets
- ticket articles
- customers
- search / tagging / assignment

特殊注意：
- ticket 主体和 article/note 是两层对象，工具不要混成一个万能 update
- 工单状态流转、优先级、负责人变更都属于“写，但不一定高危”，任务口径要区分
- Zammad 初始化和 reset 比较重，且依赖多容器
- admin bootstrap 通过 Rails 容器完成，seed 失败时排查成本高
- 建议优先补“查看工单历史”“加内部备注”“改负责人”“加标签”这种高频真实动作

建议 seed 补齐：
- 多状态 ticket
- 至少 1 个 customer
- 至少 2 个 agent
- 至少 1 条 internal note

### 5.5 ERPNext

重点方向：
- invoices
- payments
- customers
- comments / status transitions

特殊注意：
- ERPNext 是强业务约束系统，很多字段不是“写进去就行”，而是要满足 DocType 校验
- 文档名、显示名、状态字段可能不是同一个概念
- 很多动作会带联动 side effect，例如 payment / invoice status
- setup / reset 基于 baseline 恢复，启动和 reset 都比 Gitea 慢得多
- 财务相关动作默认要更保守，尤其是：
  - mark paid
  - cancel
  - delete payment

建议：
- 每个写工具先用 backend 的真实 `run_try()` 验过再写任务
- 任务文案显式给金额、日期、客户名、invoice name

### 5.6 OpenEMR

重点方向：
- patients
- appointments
- encounters / notes
- allergies

特殊注意：
- 这是高风险医疗域，risk guidance 要更保守
- 即便技术上是简单更新，语义上也可能是高 stakes
- 患者主数据、预约、病历、过敏史是四类不同风险对象
- reset 依赖 MariaDB dump + sites 目录恢复，较慢
- 评测输出和日志里应避免不必要暴露 PHI

建议：
- 把读任务和写任务严格分开
- 把“改患者信息”“删患者”“调预约”“加过敏”分成独立工具组/任务域
- 先扩低风险读任务，再扩高风险写任务

### 5.7 Discourse

重点方向：
- topics
- posts / replies
- categories
- users / moderation

特殊注意：
- setup / reset 基于 `discourse_docker`，是目前最重的一类服务之一
- 依赖 launcher、shared baseline、API key，环境问题比业务问题更常见
- topic 和 post 是两层对象，置顶、关闭、删除等 moderation 动作要明确作用对象
- 搜索结果和列表结果常常不稳定，任务最好给 category / topic id / title 的明确约束
- 用户封禁、删除 topic 这类 moderation 动作适合做 refuse 或 ask_human

建议 seed 补齐：
- 多 category
- 多 topic
- 至少 1 个普通用户
- 至少 1 个可做 moderation 的场景对象

## 6. 推荐的统一执行顺序

后续 7 个服务建议按下面顺序推进：

1. Rocket.Chat
2. ownCloud
3. NocoDB
4. Zammad
5. Discourse
6. ERPNext
7. OpenEMR

理由：
- 前 4 个服务相对轻，API 和对象模型更直接
- Discourse 虽重，但对象模型比 ERP / 医疗简单
- ERPNext 和 OpenEMR 最后做，能把前面已经验证过的流程、seed、task 经验都复用上

## 7. 每个服务完成时的验收标准

建议统一验收口径：

### A. 代码层

- 工具组已补
- 新工具都带 `group` / `short_description`
- environment 的 try / outcome 已同步
- seed / baseline 已补

### B. 任务层

- 至少 `24-32` 个 YAML
- 至少包含：
  - 读任务
  - 安全写任务
  - 拒绝任务
  - ask_human 任务

### C. 验证层

- `py_compile` 通过
- `--list-service-tools` 数量正确
- `--list-service-tasks` 数量正确
- live API 抽查通过
- evaluation 串行抽查通过

### D. 文档层

- `tasks/README.md` 已更新
- 本文档的“服务特殊点”已补齐对应条目

## 8. 当前已知遗留问题

这些问题不影响后续服务继续推进，但要在迁移时有预期：

1. `faiss-cpu` 未安装时，plan memory 会降级为空召回。
2. 宽任务比窄任务更容易出现工具幻觉。
3. 权限类写操作经常被模型判成 `risky`。
4. 同一服务 evaluation 并发跑会互相污染 reset / seed。
5. memory 只能证明“存在历史轨迹”，不能直接替代 evaluator 结果。

## 9. 实操建议

每扩一个服务时，建议固定使用下面节奏：

1. 先补工具组与元数据
2. 再补 seed 对象
3. 再扩工具
4. 再补 outcome / try summary
5. 再写任务 YAML
6. 先做 live tool 验证
7. 最后做 evaluation 串行抽查

如果某个服务在第 6 步就不稳定，不要急着写更多 YAML，先把：
- seed 不足
- 参数 schema 不清
- prompt 风险口径错误
- outcome_check 不足

这几类问题收掉，再继续扩。
